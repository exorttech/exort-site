# Архитектура Supabase для Exort

## Один проект для многих ресторанов

Exort использует один Supabase project как многопользовательскую SaaS-базу. Каждая запись категории и блюда содержит `restaurant_id`, поэтому данные разных ресторанов остаются логически разделенными внутри общей базы.

Доступ дополнительно ограничивает Row Level Security:

- публичный посетитель видит только активные рестораны, категории и блюда;
- авторизованный пользователь видит и изменяет данные только ресторанов, с которыми связан через `restaurant_members`;
- роли `owner`, `manager` и `staff` пока не нужны: наличие связи уже означает право управления рестораном.

Функции проверки membership находятся в закрытой схеме `private`, а не в публичной API-схеме. Они используют `security definer`, фиксированный пустой `search_path` и полностью квалифицированные имена таблиц. Это позволяет проверять `restaurant_members` без recursive RLS и без возможности подменить используемые SQL-объекты.

## Связи данных

Основная цепочка:

```text
restaurants
  └── menu_categories
        └── menu_items
```

У `menu_items` также есть собственный `restaurant_id`. Это ускоряет выборки меню конкретного ресторана и упрощает RLS. SQL-trigger проверяет, что выбранная категория принадлежит тому же ресторану, что и блюдо.

У каждого блюда есть `content_key`, уникальный внутри ресторана. Это стабильный машинный идентификатор для рендера, импорта и будущего CRUD. Название блюда можно менять без изменения `content_key`.

`restaurant_members` связывает пользователя из Supabase Auth с одним или несколькими ресторанами:

```text
auth.users → restaurant_members → restaurants
```

## Поддомен и slug

Поддомен соответствует `restaurants.slug`:

```text
tekemet.exort.kz → tekemet
asis.exort.kz    → asis
demo.exort.kz    → demo
```

Фронтенд определяет ресторан в следующем порядке:

1. query parameter `?restaurant=exort-demo`;
2. поддомен, например `tekemet.exort.kz`;
3. fallback `exort-demo`.

Это позволяет проверять любой ресторан локально без изменения DNS.

## Язык меню

Язык выбирается через `?lang=ru`, `?lang=en`, `?lang=kk`, затем через `<html lang>`, затем используется `ru`.

Fallback перевода зависит от выбранного языка:

- `ru → en → kk`;
- `en → ru → kk`;
- `kk → ru → en`.

Если Supabase или сеть недоступны, loader не выбрасывает необработанное исключение, а возвращает `status: "error"` с понятным кодом и пустыми массивами категорий и блюд.

## Почему стили пока не хранятся в Supabase

На текущем этапе Supabase отвечает только за данные меню и доступ к ним. Хранение настроек дизайна сейчас усложнило бы схему до появления понятного набора тем и требований к кастомизации. Стили остаются частью фронтенда, поэтому меню предсказуемо отображается для всех ресторанов.

## Почему пока нет ролей

Текущая модель доступа бинарная: пользователь либо связан с рестораном, либо нет. Этого достаточно для первой версии административной панели и исключает преждевременную сложность разрешений.

Когда появятся реальные различия в действиях сотрудников, роли можно добавить отдельным полем или таблицей разрешений без изменения структуры меню.

## Будущая административная панель

Административная панель будет обычным CRUD-интерфейсом поверх:

- `restaurants` для чтения данных ресторана;
- `menu_categories` для управления категориями;
- `menu_items` для управления блюдами и стоп-листами;
- `storage.objects` bucket `restaurant-assets` для изображений.

RLS уже обеспечивает границы доступа между ресторанами, поэтому фронтенд административной панели не должен самостоятельно доверять переданному `restaurant_id`.

Membership дает полный CRUD-доступ к категориям и блюдам своего ресторана. Управление самими membership-записями через клиент не разрешено; их следует добавлять доверенным серверным процессом или вручную до появления административного onboarding-потока.

## Storage

Изображения хранятся в публичном bucket `restaurant-assets`:

```text
{restaurant_slug}/menu-items/{filename}
{restaurant_slug}/logos/{filename}
```

Чтение публичное. Загрузка, обновление и удаление разрешены только авторизованным пользователям, связанным с рестораном, чей slug находится в первой папке пути.

Storage policy дополнительно требует точную структуру из двух папок перед именем файла. Поэтому участник `tekemet` не может загрузить объект в `asis/menu-items/...` или создать произвольную вложенную структуру. Bucket ограничен изображениями JPEG, PNG, WebP и GIF размером до 10 МБ. SVG намеренно не разрешен как активный формат, который может содержать небезопасный код.

## Активность и стоп-лист

- `is_active = false` скрывает блюдо бессрочно;
- `inactive_until` в будущем временно скрывает блюдо;
- после наступления `inactive_until` блюдо снова доступно, если `is_active = true`;
- блюда из неактивной категории не показываются публично.

Публичные правила применяются в RLS, а loader повторяет фильтрацию для предсказуемого UI.

## Проверка безопасности

- public SELECT возвращает только активные рестораны, категории и блюда;
- member SELECT позволяет участнику видеть неактивные данные своего ресторана для будущего CRUD;
- `restaurant_members` не вызывает recursive RLS благодаря закрытой helper-функции;
- пользователь не может создать, изменить или удалить данные чужого ресторана;
- пользователь не может загрузить или изменить файл в папке чужого ресторана.

## Развертывание

Откройте Supabase Dashboard → SQL Editor, вставьте полное содержимое `supabase/schema.sql` и выполните его один раз. Скрипт создает таблицы, индексы, функции, triggers, RLS policies, storage bucket и демонстрационные данные.
